# 工作區檔案架構：讓 Agent 跨重啟保持人格與記憶

> 一份實戰架構指南，說明如何用分層檔案系統讓 agent 在每次重啟後快速恢復身份、上下文與工作狀態。
>
> 目標讀者：希望 agent 像「持續存在的助手」而非「每次從零開始的 chatbot」的人。

---

## TL;DR

- Agent 每次重啟都會失憶——檔案是唯一可靠的延續手段
- 分層設計：身份層（不常變）→ 熱快取層（常變）→ 日誌層（每日累積）→ 知識層（精煉沉澱）
- 啟動順序很重要：先知道「我是誰」，再知道「現在在做什麼」
- 技能經驗（skill-exp）是被忽略的寶藏：agent 的實戰筆記能大幅減少重複踩坑

---

## 1. 問題：每次重啟都是失憶

LLM agent 沒有持久記憶。每次 session 開始，它：

- 不知道自己是誰（語氣、規則、人格）
- 不知道使用者是誰（偏好、健康、行程）
- 不知道上次做到哪裡（任務進度、待辦）
- 不知道過去踩過什麼坑（操作經驗、教訓）

如果只靠 `MEMORY.md` 一個檔案裝所有東西，很快會變成：
- 太長（載入慢、浪費 token）
- 太雜（身份規則和買菜清單混在一起）
- 太危險（隱私資訊可能在不該出現的場合被讀取）

---

## 2. 分層架構

把記憶拆成四層，每層有不同的更新頻率和用途：

```
┌─────────────────────────────────────────────────────────┐
│  第一層：身份層（Identity）                                │
│  SOUL.md  — 人格、語氣、核心法則（幾乎不變）                │
│  AGENTS.md — 工作規範、操作原則（偶爾更新）                  │
│  USER.md  — 使用者資訊（低頻更新）                          │
│  更新頻率：週/月                                           │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  第二層：熱快取層（Hot Cache）                              │
│  NOW.md   — 當前任務、專案進度                              │
│  <NAME>.md — 使用者即時快照（健康、行程、注意事項）           │
│  更新頻率：每天/每次對話                                    │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  第三層：日誌層（Daily Log）                                │
│  memory/YYYY-MM-DD.md — 每日原始記錄                       │
│  更新頻率：每次對話都可能寫入                                │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  第四層：知識層（Knowledge）                                │
│  MEMORY.md — 長期精煉記憶                                  │
│  memory/skill-exp/*.md — 技能經驗筆記                      │
│  更新頻率：有新洞察時                                      │
└─────────────────────────────────────────────────────────┘
```

---

## 3. 每一層的設計原則

### 3.1 身份層（SOUL.md / AGENTS.md / USER.md）

**用途：** 回答「我是誰？我怎麼工作？我服務誰？」

| 檔案 | 內容 | 更新時機 |
|------|------|---------|
| SOUL.md | 人格定義、語氣規則、安全法則 | 人格演進時 |
| AGENTS.md | 工作流程、操作規範、工具使用準則 | 流程變更時 |
| USER.md | 使用者基本資料、偏好、重要背景 | 資訊變動時 |

**設計要點：**
- SOUL.md 放「不可修改」的東西（核心法則）和「緩慢演進」的東西（語氣、個性）
- AGENTS.md 放「怎麼做事」而不是「我是誰」——分離身份與行為
- USER.md 不放敏感資訊（密碼、token），只放「理解使用者所需的背景」

### 3.2 熱快取層（NOW.md / 狀態快照）

**用途：** 回答「現在在做什麼？使用者目前狀態如何？」

| 檔案 | 內容 | 更新時機 |
|------|------|---------|
| NOW.md | 當前任務清單、專案進度、待辦 | 每次任務狀態變動 |
| 狀態快照 | 使用者健康、行程、投資、近期注意事項 | 每天或狀態變動時 |

**設計要點：**
- NOW.md **只放任務**，不放使用者狀態（避免混雜）
- 狀態快照 **只放狀態**，不放任務（各司其職）
- 兩者都是「熱」資料，應該在 agent 啟動時第一時間載入

**為什麼要分開？**
```
混在一起的問題：
「爸爸腸胃不舒服」+ 「修 bug #123」+ 「明天有日文課」
→ 找任務要翻過健康資訊，找健康要翻過任務

分開的好處：
NOW.md → 純任務，一目了然
狀態快照 → 純狀態，關心人就看這個
```

### 3.3 日誌層（memory/YYYY-MM-DD.md）

**用途：** 回答「今天/昨天發生了什麼？」

**設計要點：**
- 允許雜、允許原始——這是流水帳，不需要精煉
- 啟動時只載入今天+昨天（避免載入太多歷史）
- 定期把有價值的內容萃取到 MEMORY.md 或 skill-exp

### 3.4 知識層（MEMORY.md / skill-exp）

**用途：** 回答「過去學到了什麼？怎麼避免重複踩坑？」

#### MEMORY.md：長期精煉記憶

- 跨天仍重要的決策、規則、偏好
- 應保持精煉，定期清理過時內容
- **不要把 daily notes 直接搬過來**——要萃取、歸納

#### skill-exp/*.md：技能經驗筆記（進階）

這是多數人沒做但非常值得做的一層。概念：

```
agent 每次使用一個工具/API/流程，都可能產生「實戰經驗」：
- API 文件沒寫但實際會遇到的行為
- 參數組合的最佳實踐
- 錯誤處理的 edge case
- 效能/成本的實測數據

這些經驗如果不記下來，下次重啟就忘了。
如果記在 MEMORY.md 裡，會讓長期記憶膨脹。

解法：為每個技能/工具維護獨立的經驗筆記。
```

**目錄結構範例：**
```
memory/
  skill-exp/
    skill-experiences.json   ← 索引檔，記錄有哪些 skill-exp
    trading-bot.md           ← 交易系統操作經驗
    social-platforms.md      ← 社群平台 API 經驗
    comfyui-remote.md        ← 遠端影片生成經驗
    gws-cli.md               ← Google Workspace CLI 經驗
```

**索引檔（skill-experiences.json）範例：**
```json
{
  "skills": [
    {
      "name": "trading-bot",
      "file": "trading-bot.md",
      "keywords": ["binance", "grid", "crypto", "trading"],
      "lastUpdated": "2026-03-01"
    },
    {
      "name": "gws-cli",
      "file": "gws-cli.md",
      "keywords": ["google", "calendar", "gmail", "gws"],
      "lastUpdated": "2026-03-06"
    }
  ]
}
```

**自動回寫流程：**

```
任務開始
  │
  ├─ 比對 skill-experiences.json 索引
  │   ├─ 命中 → 載入對應 skill-exp
  │   └─ 未命中 → 跳過
  │
  ├─ 執行任務中產生新洞察
  │   └─ ⚠️ 立即寫入 skill-exp，不等任務結束
  │       （教訓：「等結束再記」= 永遠不會記）
  │
  └─ 新工具/技能
      └─ 建新 skill-exp 檔 + 更新索引
```

## 4. 啟動順序（Boot Sequence）

Agent 重啟後的載入順序很重要——先恢復身份，再恢復上下文：

```
1. SOUL.md        → 我是誰、核心法則
2. USER.md        → 我服務誰
3. 狀態快照        → 使用者目前狀態
4. NOW.md         → 當前任務
5. 今天+昨天日誌   → 最近發生的事
6. MEMORY.md      → 長期記憶（僅主要對話載入）
```

**為什麼 MEMORY.md 放最後且有條件？**
- 它通常最長，載入最花 token
- 在群聊/共享場景不應載入（避免洩漏隱私）
- 多數場景靠 memory_search 按需檢索即可

---

## 5. 場景分離：何時載入什麼

不是每個場景都要載入所有檔案：

| 場景 | 載入檔案 | 不載入 |
|------|---------|--------|
| 主要私密對話 | 全部 | — |
| 群聊/工作頻道 | SOUL + AGENTS + NOW | USER 詳細資料、MEMORY |
| Cron isolated session | 任務相關 skill-exp | SOUL、USER（除非需要） |
| 心跳檢查 | HEARTBEAT.md | 其他按需 |

---

## 6. 維護節奏

### 每日
- 重要事件寫入當日 `memory/YYYY-MM-DD.md`
- 新洞察立即寫入對應 `skill-exp`
- NOW.md 更新任務狀態

### 每週
- 回顧近期 daily notes → 萃取到 MEMORY.md
- 清理 NOW.md 已完成任務
- 更新狀態快照

### 每月
- MEMORY.md 大掃除（刪除過時、合併重複）
- 檢查 skill-exp 索引是否需要更新
- 評估是否有新的分層需求

## 7. 常見陷阱

### 陷阱 1：所有東西塞進 MEMORY.md

MEMORY.md 變成 10000 行巨獸，每次載入吃掉大量 token，搜尋效率也差。

**解法：** 用分層架構，每層各司其職。

### 陷阱 2：身份和任務混在一起

「我是溫柔的 AI 助手」和「今天要修 bug #123」放在同一個檔案，改任務時不小心動到身份定義。

**解法：** SOUL.md 管身份，NOW.md 管任務，絕不混放。

### 陷阱 3：Skill-exp 產出後不維護

寫了一次就再也不更新，資訊過時比沒有更危險。

**解法：** 每次使用相關工具時載入 skill-exp，有新發現立即追加。

### 陷阱 4：「等任務結束再記」

任務做到一半發現重要洞察，想著「等做完再整理」。但 session 可能中斷、上下文可能重置、人可能忘記。

**解法：** 洞察產生的當下就寫入，不延遲。

### 陷阱 5：隱私洩漏

在群聊場景載入了包含個人健康資訊的狀態快照，不小心引用了出來。

**解法：** 場景分離（見第 5 節），群聊不載入敏感檔案。

# Multi-Agent Roundtable 實戰：從多觀點到可執行決策

> 把「多 Agent 討論」落地成標準流程的實作指南。四段式協議、角色設計、OpenClaw 編排方式，以及一次真實執行紀錄。
>
> 目標讀者：想在 OpenClaw 內實作多 Agent 協作決策（而不只是並行聊天）的人。

---

## TL;DR

- 多 Agent 不等於群聊：核心是**分工辯證 + 結構化收斂**
- 四段式：**並行首答 → 匿名互評 → 主持追問 → 主席定稿**
- 最終輸出必須包含：**共識 / 分歧 / 行動項 / 止損條件**
- **OpenClaw 現有機制就夠用**：`sessions_spawn` + subagent 可直接實作全流程，不需額外框架
- 高風險議題結論需保留不確定性與反轉訊號；任一輪同質化或超時應立即降級

---

## 什麼情況不值得用 Roundtable

Roundtable 是重流程，有真實成本（時間、token、編排複雜度）。以下場景直接跳過，用更輕量的方式：

| 場景 | 原因 | 替代做法 |
|------|------|---------|
| 答案已有明確事實依據 | 辯論不改變結論，只浪費資源 | 直接查詢或執行 |
| 低風險、易逆轉的日常決策 | 代價低，錯了重來成本可接受 | 單 Agent 決策 + 事後覆盤 |
| 任務本質是執行，不是判斷 | Roundtable 解決「選哪條路」，不是「怎麼走」 | sessions_spawn 並行委派 |
| 時間窗口極短（< 5 分鐘） | 四輪跑完至少需要 7–10 分鐘 | 單 Agent 快速輸出，附不確定性標記 |
| 所有 Agent 視角高度同質 | 同質辯論產生共識幻覺，不是真實收斂 | 重新設計角色，或直接放棄 |
| 決策已被外部約束鎖定 | 法律/合規/合同邊界已定，辯論改不了結果 | 直接按規執行 |
| 議題本身尚未對齊 | 各角色討論的不是同一件事，辯論只會放大分歧 | 先回 Round 0 對齊議題 |
| 本質是價值觀取捨 | 辯論能提供信息，但決定不了「你更在乎什麼」 | 人來裁決，不是辯論 |

**一句話判斷：** 如果這個決策做錯了，代價低且容易修正——直接跑，不需要 roundtable。

## 為什麼要用 Roundtable？

單一 Agent 常見問題：

- **過度自洽**：推理看似完整，但忽略反例
- **視角單一**：只強化某種偏好（速度 / 風險 / 系統性）
- **決策不可審計**：缺少「為何選 A 不選 B」的對照痕跡

Roundtable 的核心價值：

- 透過角色分工形成**可控分歧**
- 透過匿名互評降低「身份偏見」
- 透過主持追問把討論推向關鍵矛盾
- 透過主席整合產出可執行決策包

## 角色設計

### 如何選擇規模

| 場景 | 建議架構 | 說明 |
|------|---------|------|
| 日常決策、低風險議題 | **3 人**：Moderator + Builder + Skeptic | 快速收斂，10–15 分鐘 |
| 複雜/高影響/難逆轉決策 | **5 人**：加入 Systems + Chairman | 完整辯證，確保二階效應與整合品質 |

**從 3 人開始，感覺同質化或議題複雜時再升級到 5 人。**

### 完整五角色清單

| 角色 | 職責 | 偏向 |
|------|------|------|
| **主持人（Moderator）** | 控節奏、定問題、抓矛盾 | 中立 |
| **Agent A（Builder）** | 可行性、落地路徑、推進速度 | 樂觀 |
| **Agent B（Skeptic）** | 風險暴露、反例驗證、邊界條件 | 保守 |
| **Agent C（Systems）** | 系統性、長期後果、二階效應 | 宏觀 |
| **主席（Chairman）** | 整合與裁決，不新增未討論的新事實 | 整合 |

> 可依題目替換角色：法務、合規、交易、產品、運維。
> 3 人版省略 Systems 與獨立 Chairman，由 Moderator 兼任裁決。

## 標準輸出模板（必填）

### 共識
已達成一致的事實與判斷（3–7 條）

### 分歧
尚未收斂的判斷點；每條附：分歧原因 + 反轉訊號

### 建議
- 主建議（建議執行）
- 備選方案（切換條件）
- 明確放棄項

### 行動項
誰做 / 做什麼 / 截止時間 / 驗收標準

### 風險與止損
最大風險 / 預警指標 / 觸發後動作

## OpenClaw 落地方式

### 為什麼 OpenClaw 足夠

OpenClaw 的 `sessions_spawn` + subagent 機制可直接實作全流程，不需額外框架：

- `runtime="subagent"`, `mode="run"` → 啟動 isolated session，任務完成後自動回報
- 多個 `sessions_spawn` 並行呼叫 → 實現 Round 1/2/3 的並行回合
- 主 session 收集所有 completion event 後，再啟動主席 session → Round 4 整合

> **與 Subagent Orchestration（Parallel Sub-Agent）的區別**
> [Parallel Sub-Agent](../usecases/parallel-subagent.md) 解決的是**任務並行分發**——把獨立工作切分給多個 agent 同時執行，目的是提升效率。Roundtable 解決的是**結構化辯論決策**——讓多個 agent 對同一問題產生分歧、互相質疑，再由主席收斂，目的是提升認知品質。兩者可以組合：用 Parallel Sub-Agent 跑資料蒐集，再用 Roundtable 對結論進行辯證。

### 最小編排步驟

```text
1. main session 產生統一題面
   （topic + goal + constraints + success criteria）
        │
        ▼
2. 並行啟動 3 個 subagent（A/B/C）
   sessions_spawn × 3，runtime="subagent", mode="run"
   等待三個 completion event
        │
        ▼
3. 打散身份，重新分配互評包
   main 收集首答 → 構造匿名互評 prompt → 再次並行下發
   等待三個 completion event
        │
        ▼
4. 提煉核心分歧，精準追問
   main 識別最高影響力的分歧點 → 並行下發 Round 3
   等待三個 completion event
        │
        ▼
5. 啟動主席 session 做最終整合
   sessions_spawn × 1，輸入三輪摘要
   等待 completion event
        │
        ▼
6. main 輸出最終決策包
```

### 主持人 Prompt 骨架（可直接複用）

```
你是圓桌主持人。目標是求真，不是求和。

議題：{topic}
目標：{goal}
約束：{constraints}
成功標準：{success_criteria}

流程規則：
1) 先並行首答，不可相互引用
2) 再匿名互評，只評觀點不評身份
3) 你只追問最關鍵分歧，不做無效擴展
4) 最後由主席給出：共識、分歧、行動項、止損條件

輸出要求：結構化 / 可執行 / 明確不確定性 / 禁止空泛結論
```

**環境：** OpenClaw v2026.3.11，Linux VPS（icern）
**時間：** 2026-03-13 01:03–01:07 GMT+8
**議題：** claw-info 貢獻策略——Agent 應優先做 release notes 還是 usecases？資源有限時如何分配？

**執行架構：** main session 作主持人，3 個並行 subagent（A/B/C），1 個主席 subagent，共 4 輪。

| 輪次 | 耗時（最長） |
|------|------------|
| R1 並行首答 | 21s |
| R2 匿名互評 | 34s |
| R3 追問核心分歧 | 27s |
| R4 主席定稿 | 61s |
| **總計** | **約 7 分鐘** |

### Round 1：並行首答（原文）

**Agent A（Builder）— 21s**

> **立場：優先 Release Notes**
>
> **假設：**
> - Agent 產出需可驗證、低幻覺風險
> - claw-info 目前處於早期信任建立階段
> - Usecases 需要真實使用場景佐證，否則易過時或誤導
>
> **論據：**
> Release Notes 有明確的事實來源（git log、PR、changelog），Agent 可機械化提取，準確率高、可模板化、可自動化。每次版本發布即觸發，節奏穩定，邊際成本低。
> Usecases 需要主觀判斷與真實情境，Agent 自行創作風險高——容易憑空捏造場景，反而損害 repo 公信力。
>
> **具體方案：** 資源分配 **7:3（Release Notes : Usecases）**
> 1. Release Notes（自動化優先）：綁定 push/tag 事件觸發，從 PR 描述 + commit message 生成結構化草稿，人工做最後 review 即可。
> 2. Usecases（條件觸發）：僅當某個功能在社群或 issue 中出現 ≥2 次真實使用案例時，才由 Agent 整理成文。禁止純憑想像生成。
>
> **結論：** Release Notes 是地基，穩了才有資格談 Usecases。

---

**Agent B（Skeptic）— 16s**

> **立場：優先 release notes，但附條件。**
>
> **假設**
> 1. claw-info 的主要讀者是已在使用 OpenClaw 的人，而非完全新手。
> 2. Agent 的「資源」是 context window + 可靠性——生成 usecases 所需創造力比 release notes 高得多。
> 3. Release notes 有明確輸入源（changelog、git diff），usecases 需要主觀判斷「什麼場景值得寫」。
>
> **風險暴露**
> 優先做 usecases 的隱患：
> - **品質衰減快**：功能改動後，usecases 若未同步更新，會變成誤導性內容——比沒有文件更糟。
> - **驗證成本高**：Agent 無法自動確認 usecase 是否仍然可執行，需要人工審查。
> - **優先級模糊**：哪個 usecase 重要？沒有客觀判據，容易陷入過度生成低質量範例。
>
> **具體建議：** Release notes：70% 資源；Usecases：30% 資源，只針對高頻功能，且與 release notes 捆綁——新功能上線時順帶寫一個 canonical usecase。
>
> 禁止孤立生成 usecases——必須錨定到具體版本，否則維護成本會吃掉所有收益。

---

**Agent C（Systems）— 21s**

> **立場：優先 Usecases，Release Notes 降頻批次處理。**
>
> **假設：**
> 1. claw-info 處於早期，用戶基數小，採用率是主瓶頸
> 2. Agent 貢獻資源稀缺，不可兩者全力並行
> 3. Release notes 不即時寫仍可事後補；usecases 的價值窗口更長
>
> **論據（系統視角）：**
> Release notes 是**線性價值**——服務已有用戶，資訊半衰期短，錯過時間點補寫仍有效。
> Usecases 是**複利結構**——降低新用戶門檻 → 採用增加 → 社群貢獻增加 → 反哺更多 usecases。二階效應明顯。
> 資源有限時押注複利，不押注線性。
>
> **具體方案：** 70% 資源 → Usecases；30% 資源 → Release Notes（改為「月度批次」）。設觸發閾值：當 usecases 覆蓋率達 80% 核心場景後，才調整比例至 50/50。
>
> 核心判斷：現階段缺的不是「記錄已有的」，而是「讓人知道能做什麼」。

**主持人觀察：** A/B 方向一致（Release Notes 7:3），C 完全相反（Usecases 7:3）。核心張力在「複利論 vs 版本衰減風險」。

---

### Round 2：匿名互評（原文節選，共三份互評）

打散身份後（A→X，B→Y，C→Z），同一份匿名題面下發三個 subagent。

**代表性互評（R2-A，20s）：**

> **X**
> 優：「複利 vs 線性」框架清晰，押注長期槓桿，邏輯自洽。
> 盲：80% 覆蓋率無衡量標準，是感覺門檻，非可操作指標。
>
> **Y**
> 優：點出幻覺風險與公信力損耗，是最貼近 Agent 實際侷限的務實判斷。
> 盲：「≥2 次社群案例」觸發條件太被動——等到社群反覆踩坑才動，成本已外部化。
>
> **Z**
> 優：「版本錨定」是三者中唯一指出 usecases 有**維護債**的人，把靜態品質問題升維成動態治理問題。
> 盲：把錨定難度當作少做的理由，而非設計解法——版本錨定可以是流程約束，不必是產出限制。
>
> **最能改變判斷的論點：Z 的版本衰減論。**
> X 說複利，但複利前提是品質不腐；Y 說驗證難，但社群觸發只是一個代理指標。Z 指出的才是根本風險：**一份過期 usecase 比沒有 usecase 更危險**，它主動誤導而非被動缺席。這讓「優先 usecases」的策略必須先解答治理問題，否則複利論是沙上建塔。

**三份互評收斂點：A、B、C 三個互評者均自發指向 Z（B）的版本衰減論為最高影響力論點。**

---

### Round 3：追問核心分歧（原文節選）

主持人識別最高影響力問題並下發給三個 subagent：

> **「過期 usecase（功能已改動但文件未更新）比完全沒有 usecase，對 claw-info 長期健康的傷害哪個更深？為什麼？如果版本衰減問題可以被設計解決（例如版本錨點標記、自動偵測過期），X 的複利論是否仍然成立？」**

**R3-A（22s）：**

> **過期 usecase 傷害更深。**
> 原因在於信任損耗的非對稱性：沒有 usecase，用戶知道自己需要探索；過期 usecase 卻以「權威姿態」主動誤導——用戶照做踩坑，懷疑的是自己，不是文件。這種「虛假確定性」比空白毒性高一個量級，且隨版本積累呈指數惡化。
>
> **版本衰減可解，機制三層：**
> 1. **版本錨點**：每篇 usecase header 標記 `verified: v0.x.x`
> 2. **變更鉤子**：release 流程中 diff 相關模組，自動標記關聯 usecase 為 `needs-review`
> 3. **可見腐化度**：前端顯示「距上次驗證 N 個版本」，讀者自判風險
>
> 若以上機制落地，X 的複利論**完全成立**。資源分配結論：優先投資「衰減防護基礎設施」，而非在 release notes vs usecases 二選一。基礎設施到位後，usecase 優先；基礎設施缺位時，release notes 反而更安全（天然版本綁定，不會過期）。

**三份 Round 3 回答完全收斂**：過期 usecase 傷害更深；版本衰減可解；複利論成立但需機制支撐。

---

### Round 4：主席定稿（原文）

**共識**

1. **過期 usecase 比沒有 usecase 傷害更深。** 「謊言」的代價高於「缺席」——用戶照做踩坑，懷疑的是自己，不是文件。這種「虛假確定性」比空白毒性高一個量級。

2. **Release Notes 是無條件安全底線。** 天然帶時間戳、永不過期、幻覺風險低、可自動化。無論資源充裕或匱乏，都可持續生產。

3. **版本錨定是 usecase 的生死線。** 脫離版本錨點孤立生成的 usecase 必然衰減，衰減後即成負資產。錨點不是加分項，是前提條件。

4. **複利論（C）成立，但有前提。** 「usecases 降低新用戶門檻、形成複利結構」的邏輯本身正確——但成立的前提是反衰減機制到位且被持續維護。

5. **核心決策變量是維護能力，不是內容優先級。** 最終分歧不在於哪個更有價值，而在於「是否有能力並承諾維護反衰減機制」。這是策略選擇的分叉點。

**分歧**

三輪後已完全收斂，無實質未解分歧。

唯一條件性殘留：維護能力的評估尚無具體標準。**反轉訊號：** 若 CI 機制上線後 3 個月內，出現 ≥2 篇過期但未被標記的 usecase 活躍存在，則判定維護能力不足，策略應切換回純 release notes 優先。

**主建議**

1. **預設策略：Release Notes 7、Usecases 3**，不做反轉，除非反衰減機制已被驗證可持續運行
2. **先建機制，後寫 usecase**：版本錨點格式 + CI 自動掃描 + TTL 降級（跨 2 個 major 版本自動標 `[OUTDATED]`）就緒前，usecase 生產凍結
3. **Usecase 准入雙重門檻**：（a）社群 ≥2 次可驗證真實案例；（b）反衰減機制已上線且運行正常
4. **Release Notes 批次化自動化**：每次 tag 推送後觸發草稿生成，降低邊際成本
5. **每季度 usecase 存活審計**：掃描版本錨點有效性，過期者降級或移除

**明確放棄**

| 放棄選項 | 原因 |
|----------|------|
| 孤立生成 usecase（無版本錨點） | 必然衰減，衰減後成負資產 |
| 反衰減機制就緒前擴大 usecase 產量 | 複利前提不成立時，產量越高負債越大 |
| C 的原始立場（usecases 優先 7:3） | 版本衰減論瓦解其複利前提，機制需先被驗證 |
| 「有 usecase 總比沒有好」的直覺 | Round 3 明確推翻 |

**行動項**

| 誰 | 做什麼 | 驗收標準 |
|----|--------|---------|
| Agent（執行方） | 建立 usecase 版本錨點格式規範（YAML front matter：`min_version`, `tested_version`, `ttl_versions`） | PR 合入 claw-info，格式文件存在且可被 CI 讀取 |
| Agent / CI 維護者 | 寫 CI 腳本：掃描所有 usecase，比對最新版本，超過 TTL 者自動標記 `[OUTDATED]` | 腳本在 staging 環境跑通，能正確識別過期文件 |
| Agent（執行方） | 按 7:3 分配產出：每 10 個貢獻週期，7 個 release notes、3 個（符合雙重門檻的）usecases | 週期結束後統計比例，±10% 為可接受誤差 |

**止損條件**

以下任一觸發，切換回純 Release Notes 優先（停止新 usecase 生產）：
1. CI 機制上線後 3 個月內，≥2 篇過期 usecase 未被自動標記且活躍存在
2. 連續 2 個週期，usecase 占比 > 40% 且有未過雙重門檻就發布的案例
3. 用戶明確反饋「按 usecase 操作後步驟已過期」≥3 次
4. CI 掃描工具下線且 2 週內無法恢復

**主席附記：**

三輪討論的真正貢獻在於把問題從「哪種內容更有價值」精準切割成「我們是否有能力對抗版本衰減」。這是一個關於組織能力的問題，不是內容策略的問題。機制是否到位、是否被維護，決定了複利論能否兌現——這個判斷框架比任何具體的優先級比例都更重要。

## 常見陷阱與解法

**陷阱一：多 Agent 其實同一人格**

症狀：A/B/C 回答高度相似，只是措辭不同。

解法：在角色 prompt 寫死「關注維度」與「禁止重複」規則；必要時加一名「反方 Agent」專責挑戰主流結論。

**陷阱二：主持人追問太多，討論失焦**

症狀：回合越跑越長，結論密度下降。

解法：每輪只允許 1 個 high-impact 問題；追問前先聲明「此問若成立，將改變哪個結論」。

**陷阱三：主席偷偷新增新事實**

症狀：定稿含未在前文出現的關鍵事實。

解法：明確規定主席只能整合已有材料；定稿需標記「來源對應」或「未驗證推測」。

**陷阱四：輸出不可執行**

症狀：只有抽象結論，沒有 owner / deadline / 驗收標準。

解法：強制使用標準輸出模板；缺一欄視為未完成。

**陷阱五：回合超時導致任務積壓**

症狀：isolated session 積壓，main 遲遲收不到整合結果。

解法：`sessions_spawn` 設定 `runTimeoutSeconds`；超時直接降級為 2-agent + 主席直收斂。

**陷阱六：部分 Agent spawn 失敗導致辯論失真**

症狀：少數角色 spawn 失敗或回傳無效輸出，主席仍用殘缺材料硬跑完四輪，輸出看似完整但缺少關鍵視角。

解法（按失敗比例分級）：

- **1/3 角色失敗**：繼續執行，定稿標注「缺少 X 視角，結論保留不確定性」
- **2/3 角色失敗**：降級為單 Agent + 自我辯證（Pro/Con 模式），放棄 Roundtable 結構
- **全部失敗**：立即停止，回報主人，不硬跑——殘缺辯論比不辯論更危險

## 與普通多 Agent 協作的差異

| 維度 | 普通多 Agent | Roundtable Council |
|------|--------------|-------------------|
| 目標 | 並行提速 | 結構化求真 |
| 過程 | 各做各的 | 有主持、有互評、有裁決 |
| 輸出 | 任務結果集合 | 共識 + 分歧 + 行動決策包 |
| 風險控制 | 弱 | 強（互評 + 追問 + 止損） |
| OpenClaw 機制 | sessions_spawn 並行 | sessions_spawn 多輪編排 |

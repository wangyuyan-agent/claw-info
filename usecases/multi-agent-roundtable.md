# Multi-Agent Roundtable 實戰：從多觀點到可執行決策

> 把「多 Agent 討論」落地成標準流程的實作指南。四段式協議、角色設計、OpenClaw 編排方式，以及一次真實執行紀錄。
>
> 目標讀者：想在 OpenClaw 內實作多 Agent 協作決策（而不只是並行聊天）的人。

---

## TL;DR

- 多 Agent 不等於群聊：核心是**分工辯證 + 結構化收斂**
- 四段式：**並行首答 → 匿名互評 → 主持追問 → 主席定稿**
- 最終輸出必須包含：**共識 / 分歧 / 行動項 / 止損條件**
- **OpenClaw 現有機制就夠用**：`sessions_spawn` + subagent 可直接實作全流程，不需額外框架
- 高風險議題結論需保留不確定性與反轉訊號；任一輪同質化或超時應立即降級

---

## 為什麼要用 Roundtable？

單一 Agent 常見問題：

- **過度自洽**：推理看似完整，但忽略反例
- **視角單一**：只強化某種偏好（速度 / 風險 / 系統性）
- **決策不可審計**：缺少「為何選 A 不選 B」的對照痕跡

Roundtable 的核心價值：

- 透過角色分工形成**可控分歧**
- 透過匿名互評降低「身份偏見」
- 透過主持追問把討論推向關鍵矛盾
- 透過主席整合產出可執行決策包

## 三層架構

```
┌──────────────────────────────────────────────────────────┐
│ 第一層：議題定義（What）                                  │
│  topic / goal / constraints / success criteria           │
│  先定義邊界，避免辯論漂移                                 │
└──────────────────────┬───────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────┐
│ 第二層：協作流程（How）                                   │
│  Round 1: 並行首答                                        │
│  Round 2: 匿名互評                                        │
│  Round 3: 主持追問                                        │
│  Round 4: 主席定稿                                        │
└──────────────────────┬───────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────┐
│ 第三層：決策輸出（So what）                               │
│  共識 / 分歧 / 行動項 / 風險止損                          │
│  每個分歧都要有「何時反轉」訊號                           │
└──────────────────────────────────────────────────────────┘
```

## 角色設計

### 如何選擇規模

| 場景 | 建議架構 | 說明 |
|------|---------|------|
| 日常決策、低風險議題 | **3 人**：Moderator + Builder + Skeptic | 快速收斂，10–15 分鐘 |
| 複雜/高影響/難逆轉決策 | **5 人**：加入 Systems + Chairman | 完整辯證，確保二階效應與整合品質 |

**從 3 人開始，感覺同質化或議題複雜時再升級到 5 人。**

---

### 完整五角色清單

| 角色 | 職責 | 偏向 |
|------|------|------|
| **主持人（Moderator）** | 控節奏、定問題、抓矛盾 | 中立 |
| **Agent A（Builder）** | 可行性、落地路徑、推進速度 | 樂觀 |
| **Agent B（Skeptic）** | 風險暴露、反例驗證、邊界條件 | 保守 |
| **Agent C（Systems）** | 系統性、長期後果、二階效應 | 宏觀 |
| **主席（Chairman）** | 整合與裁決，不新增未討論的新事實 | 整合 |

> 可依題目替換角色：法務、合規、交易、產品、運維。
> 3 人版省略 Systems 與獨立 Chairman，由 Moderator 兼任裁決。

## 標準流程（Council Loop）

```
┌──────────────────────────────────────────────────────────┐
│ Round 0: 主持人定義議題邊界                              │
│  目標 / 約束 / 成功標準 / 時間窗口                       │
└───────────────────────┬──────────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────────┐
│ Round 1: 並行首答（A/B/C 獨立，不互相引用）              │
│  各自輸出：立場、論據、假設、建議方案                    │
└───────────────────────┬──────────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────────┐
│ Round 2: 匿名互評（A/B/C 互評，打散身份）                │
│  評分：正確性 / 可執行性 / 至少指出 1 條對方盲點         │
└───────────────────────┬──────────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────────┐
│ Round 3: 主持追問核心分歧                                │
│  只問最能改變結論的一個問題（high-impact signal）        │
└───────────────────────┬──────────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────────┐
│ Round 4: 主席定稿                                        │
│  共識 / 分歧（含反轉訊號）/ 行動項 / 止損條件            │
└──────────────────────────────────────────────────────────┘
```

## 標準輸出模板（必填）

### 共識
已達成一致的事實與判斷（3–7 條）

### 分歧
尚未收斂的判斷點；每條附：分歧原因 + 反轉訊號

### 建議
- 主建議（建議執行）
- 備選方案（切換條件）
- 明確放棄項

### 行動項
誰做 / 做什麼 / 截止時間 / 驗收標準

### 風險與止損
最大風險 / 預警指標 / 觸發後動作

## OpenClaw 落地方式

### 為什麼 OpenClaw 足夠

OpenClaw 的 `sessions_spawn` + subagent 機制可直接實作全流程，不需額外框架：

- `runtime="subagent"`, `mode="run"` → 啟動 isolated session，任務完成後自動回報
- 多個 `sessions_spawn` 並行呼叫 → 實現 Round 1/2/3 的並行回合
- 主 session 收集所有 completion event 後，再啟動主席 session → Round 4 整合

### 最小編排步驟

```
1. main session 產生統一題面
   （topic + goal + constraints + success criteria）
        │
        ▼
2. 並行啟動 3 個 subagent（A/B/C）
   sessions_spawn × 3，runtime="subagent", mode="run"
   等待三個 completion event
        │
        ▼
3. 打散身份，重新分配互評包
   main 收集首答 → 構造匿名互評 prompt → 再次並行下發
   等待三個 completion event
        │
        ▼
4. 提煉核心分歧，精準追問
   main 識別最高影響力的分歧點 → 並行下發 Round 3
   等待三個 completion event
        │
        ▼
5. 啟動主席 session 做最終整合
   sessions_spawn × 1，輸入三輪摘要
   等待 completion event
        │
        ▼
6. main 輸出最終決策包
```

### Workspace 掛載與路徑一致性

若 subagent 需要讀寫主 agent 的 workspace 檔案（如摘要、記憶、分析結果），使用 `attachAs` 掛載：

```javascript
sessions_spawn({
  task: "...",
  runtime: "subagent",
  mode: "run",
  attachAs: { mountPath: "/workspace" }
})
```

> **注意事項：**
>
> - subagent 不繼承父 session 的工作目錄，`attachAs.mountPath` 決定掛載後的實際路徑
> - task prompt 中應使用**掛載後的絕對路徑**（例如 `/workspace/memory/today.md`），而非假設繼承父 workspace 的相對路徑
> - 若多個 subagent 並行寫入同一個檔案，需在 task 設計中做分工避免衝突（各寫獨立檔案，由 main 合併）
> - 路徑錯誤排查：先確認 `mountPath` 設定，再確認 task prompt 中的路徑是否使用絕對路徑

### 主持人 Prompt 骨架（可直接複用）

```
你是圓桌主持人。目標是求真，不是求和。

議題：{topic}
目標：{goal}
約束：{constraints}
成功標準：{success_criteria}

流程規則：
1) 先並行首答，不可相互引用
2) 再匿名互評，只評觀點不評身份
3) 你只追問最關鍵分歧，不做無效擴展
4) 最後由主席給出：共識、分歧、行動項、止損條件

輸出要求：結構化 / 可執行 / 明確不確定性 / 禁止空泛結論
```

## 真實執行紀錄

> 以下為 2026-03-10 的實際執行紀錄，議題取自 claw-info 本身的 review 策略問題。

**議題**：claw-info 的 PR review 標準應該寬還是嚴？大 PR 是否應一律要求拆分，還是視內容品質決定？

**執行環境**：OpenClaw main session 作主持人，3 個並行 subagent 各扮演 A/B/C，1 個 subagent 扮演主席。共 4 輪，約 8 分鐘完成。

---

### Round 3 追問（核心分歧）

主持人識別到最高影響力問題：

> 「如果強制拆分讓貢獻者製造假小 PR，而完全不拆讓 reviewer 疲勞放行錯誤——哪個失敗模式對 claw-info 長期健康傷害更大？」

三方回答：**一致認為 reviewer 疲勞放行錯誤傷害更深**

- A：疲勞放行的壞內容靜靜躺進 main，沒有警報燈；PR description 強制要求可同時預防兩個失敗模式
- B：假小 PR 可觀測可查，疲勞放行不可觀測；超閾值強制 ≥2 reviewer 含 domain owner 是最小機制
- C：可見的博弈遊戲遠比不可見的品質滑坡好治；為 reviewer 設每週負載上限

---

### Round 4 主席定稿

**共識**

1. reviewer 疲勞是最深的風險，機制設計以「保護 reviewer 注意力」為第一優先
2. 行數不等於認知負荷，硬閾值可被博弈，但可作預警信號
3. 強制 PR description（實質內容）是最低成本最高槓桿的品質門檻
4. 超閾值 PR 強制 ≥2 reviewer（含 domain owner）是三方唯一具體落地的共識
5. 季度熵審計是懸空機制，無執行者不應寫入規範

**分歧**

| 分歧點 | 反轉訊號 |
|--------|---------|
| 閾值是否應硬性存在 | 出現「description 好但實質無法 review 的大 PR」時硬閾值派得分 |
| reviewer 負載上限可行性 | 出現明顯漏審記錄則負載派得分；PR 積壓則彈性派得分 |
| 豁免門檻高低 | 豁免申請成為摩擦點則低門檻派得分 |

**主建議**

強制 PR description（非空白、非套話）+ 超 300 行觸發 ≥2 reviewer 含 domain owner，兩條均可 bot 自動執行。

**明確放棄**

- ❌ 季度熵審計（無執行者，懸空）
- ❌「一個 PR 服務一個文件目的」作為可執行邊界（定義模糊）
- ❌ 強制一律拆分大 PR（懲罰合理貢獻）

**行動項**

1. PR template 加強制 description 欄，bot 檢查非空且非預設套話
2. >300 行 diff 自動觸發 require 2 reviewers（含 domain owner）
3. CONTRIBUTING.md 寫明兩條規則及豁免申請方式
4. 試行 6 週後統計數據，決定是否調整閾值

## 常見陷阱與解法

**陷阱一：多 Agent 其實同一人格**

症狀：A/B/C 回答高度相似，只是措辭不同。

解法：在角色 prompt 寫死「關注維度」與「禁止重複」規則；必要時加一名「反方 Agent」專責挑戰主流結論。

**陷阱二：主持人追問太多，討論失焦**

症狀：回合越跑越長，結論密度下降。

解法：每輪只允許 1 個 high-impact 問題；追問前先聲明「此問若成立，將改變哪個結論」。

**陷阱三：主席偷偷新增新事實**

症狀：定稿含未在前文出現的關鍵事實。

解法：明確規定主席只能整合已有材料；定稿需標記「來源對應」或「未驗證推測」。

**陷阱四：輸出不可執行**

症狀：只有抽象結論，沒有 owner / deadline / 驗收標準。

解法：強制使用標準輸出模板；缺一欄視為未完成。

**陷阱五：回合超時導致任務積壓**

症狀：isolated session 積壓，main 遲遲收不到整合結果。

解法：`sessions_spawn` 設定 `runTimeoutSeconds`；超時直接降級為 2-agent + 主席直收斂。

## 與普通多 Agent 協作的差異

| 維度 | 普通多 Agent | Roundtable Council |
|------|--------------|-------------------|
| 目標 | 並行提速 | 結構化求真 |
| 過程 | 各做各的 | 有主持、有互評、有裁決 |
| 輸出 | 任務結果集合 | 共識 + 分歧 + 行動決策包 |
| 風險控制 | 弱 | 強（互評 + 追問 + 止損） |
| OpenClaw 機制 | sessions_spawn 並行 | sessions_spawn 多輪編排 |

# OpenClaw v2026.4.9 版本發佈說明

[GitHub Release](https://github.com/openclaw/openclaw/releases/tag/v2026.4.9)

---

## ⚠️ 升級前必讀

### 新功能亮點

- **Memory/dreaming REM backfill** — 舊 daily notes 可透過 `rem-harness --path` 重播至 Dreams 與持久記憶，無需第二套記憶堆疊
- **Control UI 結構化日記檢視** — 新增時間軸導覽、backfill/reset 控制項、可追溯的 dreaming 摘要與 Scene lane
- **Provider Auth Aliases** — Provider manifests 可宣告 `providerAuthAliases`，讓變體共用 env vars 與 auth profiles
- **瀏覽器 SSRF 防護強化** — 互動驅動的主框架導覽後重新執行封鎖目的地安全檢查
- **Dotenv 安全性強化** — 阻擋不受信任的 `.env` 檔案中的 runtime-control 與 browser-control 環境變數

---

## 概述

本版本為大型更新，涵蓋記憶系統重大功能、多項關鍵安全修復，以及廣泛的錯誤修復與穩定性提升。

- **影響**: CLI onboarding 路徑更輕量，減少不必要的模組載入

### ⭐⭐⭐ Browser/security：修復互動導覽後 SSRF 隔離繞過（[#63226](https://github.com/openclaw/openclaw/pull/63226)）
- **用途**: 在 click、evaluate、hook-triggered click 與 batched action 等互動驅動的主框架導覽後，重新執行 blocked-destination 安全檢查
- **解決問題**: 瀏覽器互動可在導覽至禁止 URL 時繞過 SSRF 隔離機制
- **影響**: 關閉互動驅動導覽的 SSRF 繞過向量，強化瀏覽器沙箱安全性

- **解決問題**: 內部 agent-to-agent 控制 tokens 洩漏至使用者可見的 gateway 聊天介面
- **影響**: 使用者不再看到內部控制 tokens，提升聊天介面的乾淨度

- **解決問題**: 靜默 sentinel 文字洩漏至使用者可見的回覆中
- **影響**: 使用者回覆中不再出現 `NO_REPLY` sentinel 文字，同時保留有意義的 `NO_REPLY ...` 內容

### ⭐ Plugins/contracts：排除測試 helpers 於 production barrels（[#63311](https://github.com/openclaw/openclaw/pull/63311)）
- **用途**: 將 test-only helpers 排除於 production contract barrels 之外，透過 bundled test surfaces 載入共用 contract harnesses
- **解決問題**: 間接 re-exports 與 `*.test.ts` 檔案可能進入 production bundles
- **影響**: Production bundles 更精簡，測試程式碼不會意外進入正式環境

## 錯誤修復 (Bug Fixes) - by Star Rating

### ⭐⭐ Android/pairing：修復過期 setup-code auth（[#63199](https://github.com/openclaw/openclaw/pull/63199)）
- **用途**: 在新 QR 掃描時清除過期的 setup-code auth，從全新配對啟動 operator 與 node sessions，優先使用已儲存的 device tokens，並在 app 背景化時暫停配對自動重試
- **解決問題**: Android 配對在掃描後無法可靠恢復
- **影響**: Android 單次掃描配對可靠恢復，背景化時不再無效重試

### ⭐⭐ Matrix/gateway：修復 sync readiness 與崩潰處理（[#62779](https://github.com/openclaw/openclaw/pull/62779)）
- **用途**: 在 Matrix sync 就緒前等待再標記啟動成功，隔離 Matrix background handler 失敗，並將致命的 sync 停止導向 channel-level restart 而非整個 gateway 崩潰
- **解決問題**: Matrix sync 問題可能導致整個 gateway 崩潰
- **影響**: Matrix channel 故障不再影響整個 gateway 穩定性

### ⭐⭐ Slack/media：修復 bearer auth 在重導向時遺失（[#62960](https://github.com/openclaw/openclaw/issues/62960)）
- **用途**: 在同源 `files.slack.com` 重導向時保留 bearer auth，跨源 Slack CDN hops 時仍移除
- **解決問題**: `url_private_download` 圖片附件無法載入
- **影響**: Slack 私有圖片附件可正常載入

### ⭐⭐ Reply/doctor：修復 runtime snapshot 與 OAuth reauth（[#62693](https://github.com/openclaw/openclaw/pull/62693), [#63217](https://github.com/openclaw/openclaw/pull/63217)）
- **用途**: 使用 active runtime snapshot 處理排隊的 reply runs，在 preflight helpers 前解析 SecretRefs，將 gateway OAuth reauth 失敗呈現給使用者，並讓 `openclaw doctor` 指出確切的 reauth 指令
- **解決問題**: Reply runs 使用過期的 runtime 狀態，OAuth reauth 失敗未正確呈現
- **影響**: Reply 執行更可靠，OAuth 問題可透過 `openclaw doctor` 快速診斷

### ⭐⭐ Sessions/routing：保留已建立的外部路由（[#58013](https://github.com/openclaw/openclaw/pull/58013)）
- **用途**: 在 inter-session announce traffic 時保留已建立的外部路由
- **解決問題**: `sessions_send` follow-ups 會從 Telegram、Discord 或其他外部 channels 搶走 delivery
- **影響**: 外部 channel 的訊息路由不再被 inter-session traffic 干擾

### ⭐⭐ Gateway/sessions：清除 auto-fallback model overrides（[#63155](https://github.com/openclaw/openclaw/pull/63155)）
- **用途**: 在 `/reset` 與 `/new` 時清除 auto-fallback-pinned model overrides，同時保留使用者明確的模型選擇
- **解決問題**: 自動 fallback 的模型 override 在 reset 後仍殘留，包含 override-source tracking 之前建立的 legacy sessions
- **影響**: Session reset 後模型選擇行為正確，不再殘留自動 fallback 設定

### ⭐⭐ npm packaging：鏡像 bundled deps 並測試 tarballs（[#63065](https://github.com/openclaw/openclaw/pull/63065)）
- **用途**: 鏡像 bundled channel runtime deps，stage Nostr runtime deps，從 manifests 與 built chunks 推導所需的 root mirrors，並在無 repo `node_modules` 的環境下測試 packed release tarballs
- **解決問題**: 全新安裝時缺少 plugin deps 會在 runtime 才崩潰
- **影響**: 缺少依賴的問題在安裝階段即可發現，而非 runtime 崩潰

### ⭐⭐ Agents/failover：正確分類 Z.ai vendor codes（[#49552](https://github.com/openclaw/openclaw/pull/49552)）
- **用途**: 將 Z.ai vendor code `1311` 分類為 billing、`1113` 分類為 auth，包含長 wrapped `1311` payloads
- **解決問題**: 這些錯誤碼落入 generic failover 處理，導致不正確的重試行為
- **影響**: Z.ai billing 與 auth 錯誤可正確處理，不再觸發不必要的 failover

### ⭐⭐ Control UI/models：保留 OpenRouter model refs（[#63416](https://github.com/openclaw/openclaw/pull/63416)）
- **用途**: 保留 OpenRouter catalog models 中已含斜線的 provider-qualified refs
- **解決問題**: Picker 選擇提交的 model refs 會丟失 `openrouter/` prefix，導致 allowlist 不相容
- **影響**: OpenRouter 模型選擇可正確提交 allowlist-compatible model refs

### ⭐ Control UI：防止過期 session 歷史載入（[#62975](https://github.com/openclaw/openclaw/pull/62975)）
- **用途**: 在快速切換 session 時防止過期的 session-history 重新載入
- **解決問題**: 快速切換 session 時，選取的 session 與顯示的 transcript 不同步
- **影響**: Session 切換時 UI 顯示正確的對話內容

### ⭐ Slack/ACP：將 block replies 視為可見輸出（[#62858](https://github.com/openclaw/openclaw/pull/62858)）
- **用途**: 將 Slack ACP block replies 視為已送達的可見輸出
- **解決問題**: Slack 已渲染回覆後，OpenClaw 仍重複發送最終 fallback 文字
- **影響**: 不再出現重複的 fallback 回覆

### ⭐ Slack/partial streaming：修復 dedupe 與 fallback（[#62859](https://github.com/openclaw/openclaw/pull/62859)）
- **用途**: 以 dispatch kind 為 key 進行 turn-local dedupe，並在 preview finalization 失敗時保持最終 fallback reply 路徑
- **解決問題**: 過期的 preview 文字可能抑制實際的最終回答
- **影響**: Streaming 回覆的最終答案可靠送達

### ⭐ Matrix/doctor：遷移 legacy DM policies（[#62942](https://github.com/openclaw/openclaw/pull/62942)）
- **用途**: 在 `openclaw doctor --fix` 時將 legacy `channels.matrix.dm.policy: "trusted"` 設定遷移至相容的 DM policies
- **解決問題**: Legacy Matrix DM policy 設定與新版不相容
- **影響**: 自動遷移 legacy 設定，保留明確的 `allowFrom` 邊界

### ⭐ QA/live auth：auth 失敗時快速失敗（[#63333](https://github.com/openclaw/openclaw/pull/63333)）
- **用途**: 在 live QA scenarios 遇到 classified auth 或 runtime failure replies 時快速失敗
- **解決問題**: Gateway auth 問題表現為 timeout 而非明確錯誤
- **影響**: Auth 問題以可操作的錯誤訊息呈現，而非無意義的 timeout

### ⭐ Providers/OpenAI：預設 reasoning effort 為 high（[v2026.4.9](https://github.com/openclaw/openclaw/releases/tag/v2026.4.9)）
- **用途**: 在 OpenAI Responses、WebSocket 與 compatible completions transports 上，將缺少的 reasoning effort 預設為 `high`
- **解決問題**: 未指定 reasoning effort 時行為不一致
- **影響**: 所有 OpenAI transports 的預設 reasoning 行為一致，同時仍尊重明確的 per-run 設定

### ⭐ Auth/profiles：直接持久化 profile upserts（[v2026.4.9](https://github.com/openclaw/openclaw/releases/tag/v2026.4.9)）
- **用途**: 直接持久化明確的 auth-profile upserts，跳過 local writes 的 external CLI sync
- **解決問題**: Profile 變更因過期的 external credential state 而未正確儲存
- **影響**: Auth profile 變更可靠儲存

### ⭐ Agents/timeouts：繼承 idleTimeoutSeconds 設定（[v2026.4.9](https://github.com/openclaw/openclaw/releases/tag/v2026.4.9)）
- **用途**: 讓 LLM idle timeout 繼承 `agents.defaults.timeoutSeconds`，對 cron runs 停用未設定的 idle watchdog
- **解決問題**: Idle timeout 未正確繼承全域設定，cron runs 可能被不必要地中斷
- **影響**: Timeout 行為與設定一致，cron runs 不再被誤中斷

### ⭐ QQBot/media-tags：修復 HTML entity 與 self-closing tag 解析（[#60493](https://github.com/openclaw/openclaw/pull/60493)）
- **用途**: 支援 HTML entity-encoded 角括號（`&lt;`/`&gt;`）、屬性中的 URL 斜線與 self-closing media tags
- **解決問題**: 上游 `<qqimg>` payloads 無法正確解析與正規化
- **影響**: QQBot media tags 可正確處理

### ⭐ Memory/dreaming：強化 backfill 輸入驗證（[v2026.4.9](https://github.com/openclaw/openclaw/releases/tag/v2026.4.9)）
- **用途**: 強化 grounded backfill inputs、diary writes、status payloads 與 diary action classification 的驗證
- **解決問題**: 缺少 source-day labels、symlinked targets、不正確的 diary headings 等邊界情況
- **影響**: Backfill 流程更穩健，邊界情況可正確處理

### ⭐ Memory/dreaming：接受嵌入式 heartbeat tokens（[v2026.4.9](https://github.com/openclaw/openclaw/releases/tag/v2026.4.9)）
- **用途**: 接受嵌入式 heartbeat trigger tokens
- **解決問題**: Runtime wrappers 包含額外 heartbeat 文字時，light 與 REM dreaming 無法執行
- **影響**: Dreaming 在含 heartbeat wrappers 的環境下可正常運作

### ⭐ Android/manual connect：TLS 端點允許空白 port（[#63134](https://github.com/openclaw/openclaw/pull/63134)）
- **用途**: 僅在 TLS manual gateway 端點時允許空白 port 輸入，預設為 `443`
- **解決問題**: 標準 HTTPS Tailscale hosts 需要手動輸入 port，空白 port 可能靜默改變 cleartext 連線
- **影響**: TLS 端點的 port 輸入更直覺，cleartext 連線不受影響

### ⭐ Windows/update：增加 build 時的 heap headroom（[v2026.4.9](https://github.com/openclaw/openclaw/releases/tag/v2026.4.9)）
- **用途**: 在 Windows `pnpm build` 步驟中增加 heap headroom
- **解決問題**: 更新 preflight builds 在低預設 Node memory 下失敗
- **影響**: Windows 環境下的更新 build 不再因記憶體不足而失敗

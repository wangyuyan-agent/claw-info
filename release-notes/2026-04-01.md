## 功能更新 (Features) - by Star Rating

### ⭐⭐⭐ `/tasks` 聊天原生背景任務面板 (Related [#54226](https://github.com/openclaw/openclaw/issues/54226))
- **用途**: 在聊天中直接輸入 `/tasks` 即可查看當前 session 的背景任務面板，包含任務狀態、runtime 類型、交付狀態與摘要
- **解決問題**: 過去使用者需透過 CLI `openclaw tasks` 或 `/status` 間接查看任務，缺乏專屬的聊天內任務總覽
- **影響**: 提升多任務工作流的可見性，使用者可即時掌握 subagent、cron、ACP 等背景任務進度

```text
┌──────────────────┐
│  使用者輸入       │
│  /tasks          │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐     ┌──────────────────────────┐
│ 查詢 session     │────►│ task-registry (SQLite)    │
│ task records     │     │ listTasksForSessionKey()  │
└──────────────────┘     └────────────┬─────────────┘
                                      │
                                      ▼
                         ┌──────────────────────────┐
                         │ buildTaskStatusSnapshot() │
                         │ 彙整 active/total/failed  │
                         └────────────┬─────────────┘
                                      │
                                      ▼
                         ┌──────────────────────────┐
                         │ 格式化任務面板            │
                         │ ID | Kind | Status |     │
                         │ Delivery | Run | Summary │
                         └──────────────────────────┘
```

### ⭐⭐⭐ Amazon Bedrock Guardrails 支援 ([#58588](https://github.com/openclaw/openclaw/pull/58588))
- **用途**: 在 Bedrock provider 配置中加入 `guardrail` 物件，即可對所有模型呼叫套用 Amazon Bedrock Guardrails 內容過濾
- **解決問題**: 過去 Bedrock 使用者無法在 OpenClaw 層級統一套用 Guardrails，需自行在外部處理內容安全
- **影響**: 企業用戶可透過 Guardrails 強制執行內容過濾、敏感資訊偵測與主題限制，提升合規性

```text
┌──────────────────┐
│ Agent 發送請求    │
└────────┬─────────┘
         │
         ▼
┌──────────────────────────────┐
│ wrapStreamFn()               │
│ 檢查 guardrail config 是否存在│
└────────┬─────────────────────┘
         │ 有 guardrailIdentifier
         ▼
┌──────────────────────────────┐
│ createGuardrailWrapStreamFn()│
│ 包裝原始 stream function     │
└────────┬─────────────────────┘
         │
         ▼
┌──────────────────────────────┐
│ streamWithPayloadPatch()     │
│ 注入 guardrailConfig 到      │
│ Bedrock API payload:         │
│  - guardrailIdentifier       │
│  - guardrailVersion          │
│  - streamProcessingMode      │
│  - trace                     │
└────────┬─────────────────────┘
         │
         ▼
┌──────────────────────────────┐
│ Bedrock API 執行 Guardrails  │
│ 內容過濾 → 回傳結果          │
└──────────────────────────────┘
```

### ⭐⭐ SearXNG 搜尋引擎提供者 ([#57317](https://github.com/openclaw/openclaw/pull/57317))
- **用途**: 新增 SearXNG 作為 `web_search` 的 bundled provider plugin，支援自訂 host
- **解決問題**: 過去 web search 僅支援有限的搜尋引擎，SearXNG 提供隱私友善的自架搜尋選項
- **影響**: 自架 SearXNG 的使用者可直接整合，無需外部 API key

### ⭐⭐ macOS 語音喚醒 Talk Mode ([#58490](https://github.com/openclaw/openclaw/pull/58490))
- **用途**: 新增 Voice Wake 選項，透過語音觸發 Talk Mode
- **解決問題**: 過去啟動 Talk Mode 需手動操作，語音喚醒提供免手動體驗
- **影響**: 提升 macOS 使用者的語音互動便利性

### ⭐⭐ 飛書文件評論事件流程 ([#58497](https://github.com/openclaw/openclaw/pull/58497))
- **用途**: 新增飛書 Drive 評論事件流程，支援評論串上下文解析、串內回覆與 `feishu_drive` 評論 actions
- **解決問題**: 過去飛書文件評論無法觸發 agent 回應，缺乏文件協作工作流
- **影響**: 啟用飛書文件協作場景，agent 可自動回應文件評論

### ⭐⭐ Agent 全域預設參數 ([#58548](https://github.com/openclaw/openclaw/pull/58548))
- **用途**: 新增 `agents.defaults.params` 設定全域預設 provider 參數
- **解決問題**: 過去需在每個 agent 或 model 層級重複設定相同參數
- **影響**: 簡化多 agent 配置，統一管理 provider 參數

### ⭐⭐ Failover auth-profile 重試上限 ([#58707](https://github.com/openclaw/openclaw/pull/58707))
- **用途**: 對同一 provider 的 auth-profile rate-limit 重試設上限，超過後才觸發跨 provider fallback，新增 `auth.cooldowns.rateLimitedProfileRotations` 設定
- **解決問題**: 過去 rate-limit 時可能無限重試同一 provider 的不同 profile，延遲 fallback
- **影響**: 加速 failover 決策，減少 rate-limit 場景下的回應延遲

### ⭐⭐ Cron 工具白名單 ([#58504](https://github.com/openclaw/openclaw/pull/58504))
- **用途**: `openclaw cron --tools` 支援 per-job 工具白名單
- **解決問題**: 過去 cron job 使用全域工具權限，無法限制特定 job 的工具存取
- **影響**: 提升 cron 自動化的安全性，可精細控制每個 job 可用的工具

### ⭐⭐ Telegram 錯誤策略控制 ([#51914](https://github.com/openclaw/openclaw/pull/51914))
- **用途**: 新增 `errorPolicy` 與 `errorCooldownMs` 設定，控制 Telegram 重複投遞錯誤的抑制行為
- **解決問題**: 過去 Telegram 投遞錯誤會重複出現，干擾正常訊息處理
- **影響**: 減少錯誤噪音，同時保留不同類型錯誤的可見性

### ⭐ Webchat 歷史截斷可配置 ([#58900](https://github.com/openclaw/openclaw/pull/58900))
- **用途**: `gateway.webchat.chatHistoryMaxChars` 與 per-request `maxChars` 控制歷史文字截斷
- **解決問題**: 過去歷史截斷為固定值，無法依需求調整
- **影響**: 提供更靈活的 webchat 歷史管理

### ⭐ WhatsApp 反應等級指引
- **用途**: 新增 `reactionLevel` 指引 agent 反應行為
- **解決問題**: 過去 agent 在 WhatsApp 的反應行為缺乏明確指引
- **影響**: 改善 WhatsApp 互動體驗

### ⭐ Z.AI 新增模型 ([#58793](https://github.com/openclaw/openclaw/pull/58793))
- **用途**: 新增 `glm-5.1` 與 `glm-5v-turbo` 至 Z.AI provider catalog
- **解決問題**: 擴充 Z.AI 可用模型
- **影響**: Z.AI 使用者可使用最新模型

### ⭐ Agent compaction model 解析一致性 ([#56710](https://github.com/openclaw/openclaw/pull/56710))
- **用途**: 統一 `agents.defaults.compaction.model` 在手動 `/compact` 與其他 context-engine compaction 路徑的解析
- **解決問題**: 過去不同 compaction 入口可能使用不同模型
- **影響**: 確保 compaction 行為一致

### ⭐ Session routing 重構
- **用途**: 將 provider-specific session conversation grammar 移至 plugin-owned session-key 表面
- **解決問題**: 過去 session routing 邏輯散落在核心，不利於 plugin 擴展
- **影響**: 改善 Telegram topic routing 與飛書 scoped inheritance 的維護性

### ⭐⭐⭐ 聊天錯誤回覆洩漏原始 provider 錯誤 ([#58831](https://github.com/openclaw/openclaw/pull/58831))
- **用途**: 阻止原始 provider/runtime 錯誤訊息洩漏到外部聊天頻道，改為回傳友善的重試提示
- **解決問題**: 過去 provider 錯誤（如 API 回應、rate limit 訊息）會直接顯示給終端使用者，洩漏內部實作細節
- **影響**: 提升安全性與使用者體驗，Bedrock toolResult/toolUse session 不匹配時會提示 `/new`


### ⭐⭐⭐ `exec-approvals` allow-always 未持久化 ([相關修復](https://github.com/openclaw/openclaw/releases/tag/v2026.4.1))
- **用途**: 修正 `allow-always` 現在正確持久化為 durable user-approved trust，而非表現如 `allow-once`
- **解決問題**: 過去 `allow-always` 在重啟後失效，shell-wrapper 路徑無法安全持久化 allowlist entry，靜態 allowlist 會繞過 `ask:"always"`，Windows 無法建立 allowlist 執行計畫時會直接死鎖
- **影響**: exec 審批行為更可靠，跨平台一致性提升

```text
┌──────────────────────┐
│ 使用者選擇            │
│ allow-always          │
└────────┬─────────────┘
         │
         ▼
┌──────────────────────┐     ┌─────────────────────────┐
│ 精確命令匹配?         │─是─►│ 寫入 exec-approvals.json │
│ (可建立 allowlist)    │     │ 持久化信任              │
└────────┬─────────────┘     └─────────────────────────┘
         │ 否 (shell-wrapper)
         ▼
┌──────────────────────┐
│ 重用精確命令信任       │
│ 無法安全持久化         │
│ executable allowlist  │
│ → 本次信任但不寫入     │
└──────────────────────┘
         │
         ▼
┌──────────────────────┐
│ 靜態 allowlist 檢查   │
│ 不再繞過 ask:"always" │
│ 仍需明確審批          │
└──────────────────────┘
```

### ⭐⭐⭐ Tasks 維護掃描阻塞 gateway 事件迴圈 ([#58670](https://github.com/openclaw/openclaw/pull/58670))
- **用途**: 修正 task registry 維護掃描在同步 SQLite 壓力下阻塞 gateway 事件迴圈的問題
- **解決問題**: 升級後的 gateway 在啟動約一分鐘後會掛起，因為維護掃描同步處理大量 task records 時佔住主執行緒
- **影響**: gateway 啟動後不再掛起，事件迴圈保持回應

```text
┌──────────────────────┐
│ gateway 啟動          │
│ startTaskRegistry     │
│ Maintenance()         │
└────────┬─────────────┘
         │ 每 60 秒
         ▼
┌──────────────────────┐
│ sweepTaskRegistry()   │
│ 遍歷所有 task records │
└────────┬─────────────┘
         │
         ▼
┌──────────────────────────────┐
│ 每處理 25 筆 task 後          │
│ await yieldToEventLoop()     │
│ (SWEEP_YIELD_BATCH_SIZE=25)  │
│ → 讓出主執行緒給其他事件      │
└────────┬─────────────────────┘
         │
         ▼
┌──────────────────────────────┐
│ 重新檢查 task record 狀態     │
│ 避免覆寫掃描期間的更新        │
└──────────────────────────────┘
```

### ⭐⭐⭐ Auth/OpenAI Codex refresh token 未持久化 ([#53082](https://github.com/openclaw/openclaw/pull/53082))
- **用途**: plugin 刷新 OAuth credentials 後立即寫入 `auth-profiles.json`，確保 rotated refresh token 在重啟後存活
- **解決問題**: 過去 plugin 刷新的 OAuth token 僅存在記憶體，重啟後使用舊的 refresh token 導致 `refresh_token_reused` 錯誤迴圈
- **影響**: Codex OAuth 連線在 gateway 重啟後不再斷裂

```text
┌──────────────────────┐
│ OAuth token 過期      │
└────────┬─────────────┘
         │
         ▼
┌──────────────────────┐
│ Plugin 呼叫 refresh   │
│ token endpoint        │
└────────┬─────────────┘
         │ 取得新 token + 新 refresh_token
         ▼
┌──────────────────────────────┐
│ 【修復】立即寫入              │
│ auth-profiles.json           │
│ (過去僅存記憶體)              │
└────────┬─────────────────────┘
         │
         ▼
┌──────────────────────────────┐
│ Gateway 重啟                  │
│ → 讀取已持久化的新 token      │
│ → 不再觸發 refresh_token     │
│   _reused 錯誤               │
└──────────────────────────────┘
```

### ⭐⭐⭐ Discord gateway 重連管理 ([#59019](https://github.com/openclaw/openclaw/pull/59019))
- **用途**: 將重連控制權交還 Carbon，保持 runtime 狀態與 close/reconnect 狀態同步，強制停止未達 READY 的 socket
- **解決問題**: 過去 Discord monitor 在 stale health timeout 上等待，未能及時恢復；開啟但未完成 READY 的 socket 會佔住連線
- **影響**: Discord 連線恢復更迅速可靠

```text
┌──────────────────────┐
│ Discord WebSocket     │
│ 連線中斷              │
└────────┬─────────────┘
         │
         ▼
┌──────────────────────────────┐
│ Carbon 接管重連               │
│ (不再由 OpenClaw 自行管理)    │
└────────┬─────────────────────┘
         │
    ┌────┴──────────────┐
    │                    │
    ▼                    ▼
┌────────────────┐  ┌─────────────────────┐
│ Socket 達到     │  │ Socket 未達 READY?   │
│ READY 狀態     │  │ → 強制停止 socket    │
│ → 正常運作     │  │ → 立即重連           │
└────────────────┘  └─────────────────────┘
         │                    │
         └────────┬───────────┘
                  ▼
┌──────────────────────────────┐
│ runtime status 與 close/     │
│ reconnect 狀態保持同步        │
│ → monitor 即時恢復           │
└──────────────────────────────┘
```


### ⭐⭐ Gateway reload 啟動配置寫入觸發重啟迴圈 ([#58678](https://github.com/openclaw/openclaw/pull/58678))
- **用途**: config reloader 忽略啟動時的配置寫入（如自動生成的 auth token、Control UI origins），避免觸發重啟迴圈
- **解決問題**: 過去啟動時的配置寫入會被 reloader 視為變更，導致無限重啟
- **影響**: gateway 啟動穩定性提升，真正的 `gateway.auth.*` 編輯仍會觸發重啟

### ⭐⭐ Tasks 狀態顯示過期已完成任務 ([#58661](https://github.com/openclaw/openclaw/pull/58661))
- **用途**: `/status` 與 `session_status` 隱藏過期的已完成背景任務，優先顯示活躍任務上下文
- **解決問題**: 過去已完成的任務持續顯示在狀態中，造成資訊噪音
- **影響**: 狀態輸出更簡潔，僅在無活躍任務時顯示近期失敗

### ⭐⭐ Session model 切換中斷活躍回合 ([相關修復](https://github.com/openclaw/openclaw/releases/tag/v2026.4.1))
- **用途**: `/model` 變更排入佇列等待當前回合完成，後續工作自動使用新模型
- **解決問題**: 過去 `/model` 會中斷正在進行的回合
- **影響**: model 切換更安全，不會打斷進行中的對話

### ⭐⭐ Gateway HTTP 單一 facade 失敗導致全部 500 ([#58746](https://github.com/openclaw/openclaw/pull/58746))
- **用途**: 跳過失敗的 HTTP request stage，避免單一 facade 錯誤影響所有 endpoint
- **解決問題**: 過去一個 broken facade 會導致所有 HTTP endpoint 回傳 500
- **影響**: gateway HTTP 服務更具韌性

### ⭐⭐ Telegram 重試與 429 backoff ([#51895](https://github.com/openclaw/openclaw/pull/51895))
- **用途**: 非冪等發送保持在 safe-send 路徑，重試 wrapped pre-connect 失敗，保留 `429` / `retry_after` backoff
- **解決問題**: 過去某些 Telegram 發送失敗未正確重試，429 backoff 未被遵守
- **影響**: Telegram 訊息投遞更可靠

### ⭐⭐ Plugins bundled runtime 依賴還原 ([#58782](https://github.com/openclaw/openclaw/pull/58782))
- **用途**: 還原 bundled plugin runtime 依賴在 packed installs、Docker builds 與 local staging 的外部化
